# D6 바이브코딩 핵심개념 v2 순서 정리

## 한 줄 결론
지금 PDF는 내용 자체는 좋지만, `개념 -> 어떻게 시키는가 -> 어떻게 설계하는가 -> 어떤 기술이 들어가는가 -> 어떻게 안전하게 검증하는가 -> 문서는 왜 나뉘는가` 순서로 읽히면 더 잘 들어온다.

## 추천 읽기 순서

### 1. 먼저 큰 그림 잡기
- `p.3` 바이브코딩이란
- `p.7` 챗봇 vs 에이전트
- `p.9` Codex란 무엇인가

이 3장을 먼저 보면 "지금 우리가 뭘 배우는지"가 잡힌다.

### 2. 그다음 AI가 왜 흔들리는지 이해
- `p.4` LLM 한 장 정리
- `p.5` 토큰 · 온도 · 확률
- `p.6` 환각과 검증

이 구간은 AI의 작동 원리보다 "왜 검증이 필요한가"를 이해하는 용도다.

### 3. 이제 실제로 어떻게 시킬지 배우기
- `p.10` 좋은 요청 5요소
- `p.11` 권한 · 승인 · 샌드박스
- `p.12` 폴더 · 환경 · .env
- `p.13` 스킬
- `p.14` MCP
- `p.15` Git · GitHub

이 묶음은 "AI를 업무 도구로 안전하게 쓰는 법" 파트로 보면 된다.

### 4. 그다음 에이전트 설계 감 잡기
- `p.17` instruction 7요소
- `p.18` 입력 -> 처리 -> 출력
- `p.19` 도구
- `p.20` 자동화 레벨과 HITL
- `p.21` 오케스트레이션
- `p.22` 실패 대비 설계
- `p.23` 기획 먼저

여기는 "답변 잘하는 AI"가 아니라 "일 잘하는 AI"를 만드는 설계 파트다.

### 5. 기술은 그다음에 붙이기
- `p.25` API
- `p.26` JSON
- `p.28` 프런트엔드
- `p.29~30` 백엔드 / 프런트엔드 vs 백엔드
- `p.31` Supabase
- `p.32` RAG
- `p.33` 검색 원리
- `p.27` BizRouter
- `p.35` Vercel
- `p.36` 비용 · 캐시
- `p.34` 디자인

비개발자 기준으로는 기술을 너무 일찍 넣으면 부담이 커진다. 그래서 설계 감을 잡은 뒤에 "아, 이 부품들이 여기 들어가는구나" 식으로 보는 게 좋다.

### 6. 마지막에 안전과 검증
- `p.38` 근거 · HITL · escalate
- `p.39` 검증 골든셋과 멀티에이전트 평가

실무에서는 이 파트가 아주 중요하지만, 학습 순서상 맨 뒤에 두면 앞 내용을 한 번 정리하는 효과가 있다.

### 7. 맨 끝에 문서 체계 정리
- `p.41` 왜 문서가 여러 개일까
- `p.42` 집 짓기 비유
- `p.43` 가장 헷갈리는 5쌍
- `p.44` 한 장 요약
- `p.45` 용어사전
- `p.46` 헷갈리는 개념 구분

이 파트는 본문이라기보다 "복습 자료"에 가깝다. 그래서 끝에 두는 편이 더 자연스럽다.

## 발표용으로 더 좋은 최종 목차

1. 바이브코딩이란 무엇인가
2. AI는 왜 그럴듯하지만 틀릴 수 있는가
3. 그래서 사람은 무엇을 맡아야 하는가
4. Codex로 어떻게 안전하게 일을 시키는가
5. 에이전트는 어떤 구조로 설계하는가
6. 실제 구현에는 어떤 기술 부품이 들어가는가
7. 품질과 책임은 어떻게 통제하는가
8. 프로젝트 문서는 왜 나눠서 관리하는가

## 지금 PDF 기준으로 아쉬운 순서
- `Codex(p.9)`가 조금 늦게 나온다. 비개발자 입장에서는 초반에 나오는 편이 이해가 쉽다.
- `챗봇 vs 에이전트(p.7)`는 앞쪽으로 당기는 게 좋다. 뒤에 나오는 설계 파트의 이해도가 올라간다.
- `기술 한 장씩` 파트는 내용은 좋지만 길다. 발표에서는 모두 다 설명하지 말고 `API / 프런트-백 / Supabase / RAG`만 우선 설명하는 편이 낫다.
- `문서 구분` 파트는 본문 중간보다 맨 마지막 부록처럼 읽히는 편이 더 잘 맞는다.

## 가장 쉬운 암기 순서
`개념 -> 한계 -> 요청법 -> 설계 -> 기술 -> 안전 -> 문서`

이 한 줄만 기억하면 전체 흐름이 안 헷갈린다.
